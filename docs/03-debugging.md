# 演習3: デバッグ - Playwright MCPを使ったパフォーマンス問題の特定

## 概要

この演習では、デバッグ情報が一切ないアプリに対して、GitHub Copilot と Playwright MCP を駆使して**フロントエンド側のパフォーマンス問題**を調査します。「どこが遅いか」を検証するために、Playwright MCP のブラウザ JavaScript 実行機能や・コードへのデバッグログ追加を活用します。

## 学習目標

- Playwright MCP の `browser_evaluate` でブラウザのJavaScriptを実行して情報収集できる
- `browser_console_messages` でブラウザコンソールのログを取得できる
- Copilot にフロントエンドコードを修正させてデバッグ情報を追加できる
- ネットワーク vs フロントエンドJSのどちらにボトルネックがあるかを判断できる
- パフォーマンス改善の提案ができる

## 前提条件

- アプリが起動していること（`npm run dev`）
- Playwright MCP が有効になっていること
- GitHub Copilot Chat が利用できること

---

## 背景: ダッシュボードの問題

「パフォーマンスダッシュボード」ページ（`/dashboard`）で「レポートを生成」ボタンをクリックすると、**非常に時間がかかる**という報告があります。

ユーザーからのフィードバック：

> 「レポート生成に数秒かかっていて、使いにくい。どこが遅いのか調べてほしい。」

現在のアプリには**パフォーマンスに関するログや計測コードが一切ありません**。「ネットワークが遅いのか、ブラウザ上のJS処理が遅いのか」――画面を見るだけでは判断できません。どんな手段で調査しますか？

---

## 演習手順

### ステップ1: 問題を再現する

GitHub Copilot Chat に、以下の**シナリオを達成するプロンプトを自分で書いて**入力してください。

#### 📋 達成するシナリオ

> プレイショップのダッシュボードページ（`/dashboard`）で「レポートを生成」ボタンを押し、処理が完了するまで待機した後、完了した画面のスクリーンショットを撮影する。

#### 💡 プロンプトを書くときのヒント

- どの URL から始めるか
- どのボタンを押すか
- 処理完了まで待機すること
- スクリーンショットを撮影すること

> ❓ 体感でどのくらい時間がかかりましたか？遅さの原因はどこにあると思いますか？まず仮説を立ててみましょう。

---

### ステップ2: ボトルネックを調査する

ボトルネックがどこにあるかを特定するために、**自分で調査方針を決めて、プロンプトを書いて**実行してください。

「どうすれば何が遅いかわかるか？」を考えるのがこのステップの核心です。

#### 📋 達成するシナリオ

> どの処理にどれくらい時間がかかっているかを特定する。

#### 💡 調査に使える手段（組み合わせ自由）

| 手段 | 概要 | Copilot / Playwright での操作 |
|------|------|-------------------------------|
| **ネットワーク通信時間の確認** | APIがどのくらいで応答しているか調べる | `browser_evaluate` で `performance.getEntriesByType('resource')` を実行 |
| **フロントエンドコードへのログ追加** | ブラウザで動く JS の各処理に計測ログを埋め込む | Copilot にコード修正を依頼 → `browser_console_messages` でログ取得 |
| **ソースコード調査** | フロントエンドのコードを読んで怪しい処理を探す | Copilot にファイルを読んで説明してもらう |

> ❓ どの手段を使いましたか？それぞれからどんな情報が得られましたか？

---

### ステップ3: 根本原因を特定する

収集した情報をもとに、**ボトルネックの根本原因を特定するプロンプトを自分で書いて**入力してください。

#### 📋 達成するシナリオ

> 調査で得られた結果と `src/app/dashboard/page.tsx` のコードを照らし合わせて、なぜおそいのかを具体的に説明してもらう。

---

### ステップ4: パフォーマンス分析レポートの作成

調査結果をまとめて**パフォーマンス分析レポートを作成するプロンプトを自分で書いて**入力してください。

#### 📝 レポートに含めてほしい内容

- 各処理の所要時間の表（調査で取得した値）
- ボトルネックとなっている処理とコード上の原因
- 短期・中長期的な改善提案

## 参考: デバッグに使えるAPI

### Console API（フロントエンドコードへのログ追加）

```javascript
// 処理の実行時間計測（フロントエンドコードに追加する）
console.time("ラベル名");
// ... 計測したい処理 ...
console.timeEnd("ラベル名");
```

### Performance API（ブラウザ側でAPIの通信時間を確認）

```javascript
performance.getEntriesByType("resource")
  .filter(entry => entry.name.includes("/api/"))
  .map(entry => ({
    url: entry.name,
    duration: Math.round(entry.duration),
    waiting: Math.round(entry.responseStart - entry.startTime),
  }));
```

## 実際のボトルネック（ヒント）

<details>
<summary>❗ 演習を試した後に開くことをおすすめします</summary>

ボトルネックはフロントエンド（ブラウザ側）の処理にあります。APIは高速に応答しますが、`src/app/dashboard/page.tsx` 内で表示用データの整形処理が実行されており、ここで大量の計算が発生しています。Performance APIで通信時間を確認すると「ネットワークは速い」にもかかわらず惿いが雠いことがわかり、Copilot にフロントエンドコードへの `console.time` 追加を依頼することで原因が特定できます。

</details>
